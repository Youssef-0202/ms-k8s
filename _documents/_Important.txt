II-  API Gateway usinf Spring Cloud Gateway MVC :  (to search <Functional endpoints models !!>):

Spring Cloud Gateway originally was reactive only, based on WebFlux and Netty.
Not all applications use reactive programming. Some still use the traditional, blocking Spring MVC.
So, the Spring team decided to offer two gateway implementations:
One reactive for apps built with WebFlux.
One blocking (MVC) for apps built with Spring MVC.
This allows developers to choose the gateway that best matches their app’s architecture.

V- integrate Kafka to asynchronus connection between order and notification :

  A - Kafka Microservices Architecture & Flow :
    1- Core Concepts :
       - Topic : Named category/queue where events are written (e.g., "order-placed"). It acts as a mailbox.
       - Producer : The service sending the event (order-service) via Spring's predefined 'KafkaTemplate'.
       - Consumer : The service reading the event (notification-service) via the '@KafkaListener' annotation.
       - Serializer / Deserializer :
         . Producer uses 'KafkaAvroSerializer' to encode Java objects into compact Avro binary format.
         . Consumer uses 'KafkaAvroDeserializer' to decode Avro binary packets back into Java objects.
       - Schema Registry (Port 8085) : Central validation point ensuring both services conform to the same data schema.
    2- Communication Flow :
       - OrderService creates and saves order -> Generates orderPlacedEvent (defined in .avsc schema).
       - OrderService publishes the event to topic: `kafkaTemplate.send("order-placed", orderPlacedEvent)`.
       - Spring sends response 200 immediately to frontend (decoupled execution, non-blocking checkouts).
       - NotificationService consumes event: `@KafkaListener(topics = "order-placed") public void listen(OrderPlacedEvent event)`.
       - NotificationService processes event and sends transaction confirmation email via JavaMailSender (Mailtrap).

  B - Shared Library Concept (Jar Sharing) :
    1- Definition : Instead of duplicating schema (.avsc) files and class generation configuration inside every microservice, shared types are compiled into a standalone Maven project (e.g. `event-library`), packaged into a `.jar` library, and installed in the local machine's repository cache (`.m2` cache).
    2- Implementation Steps :
       - Step 1: Create an `event-library` directory. Inside `pom.xml`, add `avro` dependency and the `avro-maven-plugin` to generate classes under `/target/generated-sources/avro`.
       - Step 2: Write schemas in `src/main/resources/avro/order-placed.avsc`.
       - Step 3: Run `mvn clean install` inside `event-library`. This compiles classes and loads the built `.jar` file directly into your local `.m2/repository`.
       - Step 4: Delete all local duplicates of schemas and generated classes under `/src/` directories of `order-service` and `notification-service`.
       - Step 5: Add a standard Maven dependency in the `pom.xml` of both producer and consumer projects :
         <dependency>
             <groupId>com.techie.microservices</groupId>
             <artifactId>event-library</artifactId>
             <version>1.0.0</version>
         </dependency>
       - Step 6: Trigger "Reload All Maven Projects" in IntelliJ. Import and use the shared classes seamlessly:
         `import com.techie.microservice.order.event.OrderPlacedEvent;`

VII- Choose Internal Communication Pattern (synchrone vs asynchrone) :

1. Sync HTTP/REST (eg. OpenFeign or Spring 6 RestClient HTTP Interface) :
   - Pros: Simple, transparent JSON logs, easy debugging with Postman, low learning curve.
   - Cons: CPU/bandwidth verbosity, slower (HTTP 1.1 + text deserialization), tight coupling.
   - Best for: Outer-facing APIs (Front to Gateway) or simple Back-to-Back links where latency is not a critical bottleneck.
   - Note for Inventory: Crucial to keep synchrone links (REST/gRPC) for checkout balance deduction (prevents overselling/race conditions on stock limits).

2. Sync gRPC (HTTP/2 + Protocol Buffers) :
   - Pros: Extremely fast (up to 10x faster than REST), compact binary payload, code contracts generated via .proto templates.
   - Cons: Higher setup complexity, browser client incompatibility (requires gRPC-Web proxy), harder debugging without tools.
   - Best for: Internal Back-to-Back microservice links in heavy load enterprise architectures.

3. Async Message Brokers (Kafka / RabbitMQ) :
   - Pros: Maximum decoupling, high availability (resilient during consumer crashes), high throughput.
   - Cons: Complex transaction patterns (requires Saga or Outbox logic), eventual consistency lag.
   - Best for: Non-critical operations (emailing, invoicing, async cache updates, database replicas updates).

VIII- Gateway Circuit Breaker & Fallback System (Resilience4J) :

1. Configuration in Routes :
   - Gateway routes use a filter to intercept network errors/timeouts :
     `.filter(CircuitBreakerFilterFunctions.circuitBreaker("productServiceCircuitBreaker", URI.create("forward:/fallbackRoute")))`
   - If target service fails, traffic is intercepted and forwarded internally to `/fallbackRoute`.

2. The /fallbackRoute Implementation (Missing by Default) :
   - If not created, client gets generic 404 blockages when service is down.
   - Fix Option A (Local Route inside Routes.java) :
     @Bean
     public RouterFunction<ServerResponse> fallbackRoute() {
         return route("fallback_route")
                 .GET("/fallbackRoute", request -> 
                     ServerResponse.status(HttpStatus.SERVICE_UNAVAILABLE)
                         .contentType(MediaType.APPLICATION_JSON)
                         .body("{\"message\": \"Service is temporarily down. Please try again later.\"}")
                 ).build();
     }
   - Fix Option B (Fallback Controller) : Custom @RestController mapping "/fallbackRoute" to return JSON.

3. The Three States of Circuit Breaker :
   - CLOSED (Default) : Traffic flows normally. Error rates/timeouts are monitored in sliding windows (e.g. 10 calls). If error rate > 50%, circuit transitions to OPEN.
   - OPEN : Prevents flooding a down service. Intercepts all traffic immediately and redirects to /fallbackRoute. Keeps this state for cool-down timer (e.g. 5s).
   - HALF-OPEN : Lets limited test calls pass (e.g. 3 calls) to verify service recovery. If ALL succeed -> transitions back to CLOSED (recovered). If any fails -> returns to OPEN state.

IX- Frontend Error Handling (Axios + Spring Boot @ControllerAdvice) :

RULE: Axios automatically throws an exception (AxiosError) for any HTTP response with status >= 400.
      Always handle errors in the component/page layer (App.jsx), NOT in the service layer (apiService.js).
      The service layer must only propagate errors with `throw error`.

1. Case 1 - Server responded with error (4xx/5xx) :
   - err.response EXISTS
   - err.response.status    → HTTP code (400, 404, 409, 500...)
   - err.response.data      → Body of the Spring Boot error response (JSON from @ControllerAdvice)
   - err.response.data.message → The actual business error message (e.g. "Product 'ps-01' not found")
   - Usage : const msg = err.response.data?.message || JSON.stringify(err.response.data);

2. Case 2 - No response received (service DOWN / network timeout) :
   - err.response is UNDEFINED
   - err.request EXISTS (request was built and sent, but no response came back)
   - err.code = "ERR_NETWORK" or "ECONNABORTED"
   - Usage : Show "Service inaccessible. Check that backend is running."

3. Case 3 - Axios configuration error (bad URL, etc.) :
   - err.response is UNDEFINED, err.request is UNDEFINED
   - err.message contains the Axios internal error (e.g. "Invalid URL")
   - Usage : Show `Erreur de configuration : ${err.message}`

4. Complete pattern to use in catch blocks (App.jsx) :
   catch (err) {
     if (err.response) {
       // SPRING BOOT MESSAGE : err.response.data.message
       const msg = err.response.data?.message || JSON.stringify(err.response.data);
       setError(`[${err.response.status}] ${msg}`);
     } else if (err.request) {
       setError("Service inaccessible. Vérifiez que le backend est démarré.");
     } else {
       setError(`Erreur de configuration : ${err.message}`);
    }
  }

X - Product Image Service & Image Storage Decoupling (MinIO + Gateway Security) :

  1 - Architecture & Database Decoupling :
      - Instead of storing binary image data directly in the MongoDB product collection, image binary persistence is delegated to a dedicated microservice `image-service` configured with MinIO (S3 compliant object storage).
      - Collection `products` in MongoDB is updated to persist only a lightweight `imageId` (UUID of the object stored in MinIO).
      - Product DTOs (`ProductRequest`, `ProductResponse`, `ProductDto`) are updated to handle the `imageId` field.

  2 - Gateway Security Bypass for Public Assets (Keycloak) :
      - In a microservices architecture secured by Keycloak/OAuth2, direct resource URLs (like <img src="..."/>) will fail with a `401 Unauthorized` status code unless the endpoint is publicly accessible.
      - Fixed by configuring the Gateway Security filter chain in `SecurityConfig.java` to make the `GET /api/image/**` routes public, while keeping write/mutation paths (`POST`, `DELETE`) protected :
        `.requestMatchers(HttpMethod.GET, "/api/image/**").permitAll()`

  3 - Frontend Integration Flow :
      - Step A (Silent Upload) : Immediately upon selecting a file in the product creation modal, the frontend uploads the binary file using `multipart/form-data` to the `image-service` via Gateway, retrieving a returned `imageId` (UUID).
      - Step B (Metadata Persist) : The final form submit maps the input form state including the `imageId` value to create the product document in `product-service`.
      - Step C (Re-rendering) : Catalog displays images by passing the stored `/api/image/{imageId}` URL inside a simple `<img>` tag, relying on standard browser automatic GET requests.
